# @ocrjs/ir

## 概要

`ir` は、ocrjs における **動的な中間表現（Intermediate Representation）スタッカー** を提供するパッケージです。

このパッケージの目的は、`infra-contract` で定義されたインターフェースを介して発行される「メッセージ（メソッド呼び出し）」をリアルタイムに記録し、メモリ上に **計算スタック（Execution Stack）** を構築することにあります。

## 設計思想：動的なスタック構築 (Dynamic Stacking)

本アーキテクチャは、静的なコード解析（プログラムを文字列として読み解くこと）を一切行いません。代わりに、JS/TS の実行プロセスそのものを「スタック構築のトリガー」として利用します。

### 1. 「記録係」としての Infra-Adapter
`ir` は `infra-contract` を実装した一つのアダプターとして機能します。ユーザーが `ITensor.add()` や `IImage.resize()` を呼び出した際、このアダプターは即座に計算を行うのではなく、「どのような操作が、どの順序で行われたか」をスタックに積み上げます。

### 2. JS/TS の柔軟性との共存
動的にスタックを積むため、ユーザーがメソッド呼び出しの間に独自の JS/TS ロジック（if 文、ループ、計算など）を挟んでいても、スタッカーはそれらの実行結果として呼ばれた「契約メソッド」を正しい順序で記録し続けることができます。

### 3. 想定外の事態へのセーフティネット
IR スタッカーが処理できない想定外のデータや複雑な操作が検知された場合、スタッカーは即座にその情報を `IErrorable` のエラー状態として記録し、従来の即時実行型アダプターと同様の安全なエラーハンドリングフローにフォールバックします。

## 最適化の三層構造

スタックされた計算の連鎖は、実体化（Materialization）のタイミングで以下の三段階の最適化を経て実行されます。

1. **MIR (Mathematical IR)**: 行列演算やテンソル演算を数学的に解析し、カーネル融合（Kernel Fusion）や並列化を行います。
2. **JIR (JavaScript-friendly IR)**: JS ランタイム（V8/JSC）の特性に合わせ、GC（ガベージコレクション）を考慮したメモリ管理やバッファの再利用をスケジュールします。
3. **RIR (Runtime IR)**: WebGPU, Node.js, Bun, Wasm といった物理的な実行環境に特化した最終命令を発行し、Python (PyTorch) を超える実行効率を目指します。

## 安定性と将来の展望

現在は JS の実行に相乗りする形をとりますが、将来的に「生 JS 処理」を含んだスタックの安定性を高めるため、専用のインターフェース（`engine.raw()` 等）を介して JS ブロックを一つのノードとして明示的に扱う仕組みを導入する可能性があります。

また、ユーザーが「レシピ」の構築中に内部データに直接アクセスすることを防ぐため、データの永続化や共有を司る `store` の仕組みを提供し、IR スタッカーが常にデータのライフサイクルを完全に把握できる環境を構築します。
